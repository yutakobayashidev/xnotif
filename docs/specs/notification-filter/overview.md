# Notification Filter

## Summary

通知タイプに基づくフィルタリング機能を `NotificationClient` に追加し、ユーザーが不要な通知を受信前に除外できるようにする。

## Background & Purpose

現状の xnotif はすべての通知を `notification` イベントとして発火する。実際のユースケースでは特定タイプ（tweet, mention 等）のみが必要なケースが大半で、ユーザーはイベントハンドラ内で自前フィルタを書く必要がある。

ライブラリレベルでフィルタリングを提供することで：

- ボイラープレートを削減し、10行以内の初期化という設計原則を維持
- フィルタ条件を `createClient` のオプションに集約し、設定の一覧性を向上
- 将来のフィルタ拡張（正規表現、ユーザー指定等）の拡張ポイントを確立

## Why Now

- v0.2.0-beta.1 が公開済みで、コアライブラリが安定した次のフェーズ
- tweets.jsonl の分析で `data.type` フィールドの構造が安定していることを確認（54件/1時間の観測で全件同一キー構造）
- ライブラリ化（P0）・npm公開（P1）が完了し、P2 エピックに着手するタイミング

## Hypothesis

- **H1**: `filter` オプションを追加すれば、ユーザーは通知ハンドラ内の条件分岐を削除でき、セットアップコードが簡潔になる

## Expected Outcome

- predicate 関数で任意の条件指定が可能（`filter: (n) => n.data?.type === "tweet"`）
- 既存の API に破壊的変更なし — `filter` 未指定時は従来通り全通知を発火
